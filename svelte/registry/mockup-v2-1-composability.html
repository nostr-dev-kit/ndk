Pre-built vs Primitives

Need a user card? Use the pre-built one. Need something custom? Compose it from primitives. Same components. Same data. Total flexibility.

Option 1: Use The Pre-built

// Drop in a complete user card <UserCard.Classic user={user} /> // Renders: avatar, name, nip05, bio, follow btn // Everything styled and wired up
7 variants available: Classic, Compact, Glass, Landscape, Neon, Portrait, Modern

Option 2: Compose From Primitives

// Build your own layout <User.Root {user}> <User.Avatar size="large" /> <User.Name class="text-2xl" /> <User.Nip05 showCheck={true} /> <!-- Skip bio, add custom content --> <div>Custom content here</div> </User.Root>
Mix, match, reorder, skip pieces. Total control.

The Power: Swap Pieces Without Breaking Anything

// Start with pre-built <UserCard.Classic {user} /> // Need custom avatar? Just override that one piece <UserCard.Classic {user}> <User.Avatar size="xl" shape="square" fallback={CustomFallback} /> </UserCard.Classic> // Everything else stays the same. Name, bio, follow button all work.

Swap Renderers, Keep The System

Want to change how mentions render? How article embeds look? Swap the renderer. The content parser doesn't care. The event card doesn't break.

Example: Custom Mention Rendering

// Default mention rendering <EventCard {event}> <Event.Content {event} /> </EventCard> // Mentions render as: @username // Want clickable avatars instead? Swap the renderer <EventCard {event}> <Event.Content {event} mentionComponent={AvatarMention} /> </EventCard> // Now mentions render with user avatars // Or build your own mention component <Mention.Modern user={mentionedUser} onClick={handleMentionClick} />
The content parser finds mentions. You decide how they render. Separation of concerns that actually works.

Reactions: More Complex Than You Think

A reaction button seems simple. Until you need +1, emoji picker, NIP-30 custom emojis, showing who reacted, rendering reactions in feed vs article vs highlight...

Basic +1 Reactions

<ReactionButton.Basic event={event} /> // Just like/unlike // Shows count

With Emoji Picker

<ReactionButton event={event} picker={true} /> // Click = emoji picker // NIP-25 reactions

Slack-Style Display

<Reaction.SlackEmoji event={event} /> // Groups by emoji // Shows counts // Click to add yours

Show Who Reacted

<Reaction.Avatars event={event} /> // Avatar stack // Hover for names // Click to see all

NIP-30 Custom Emojis

<ReactionButton event={event} customEmojis={true} /> // Renders image emojis // In reactions AND content

Context-Aware

<Reaction.Display event={event} context="article" /> // Different layout for // notes vs articles

The Hard Part: Making Them All Work Together

// Reactions with custom NIP-30 emojis <EventCard {event}> <Event.Content customEmojiRenderer={CustomEmojiImage} /> <Event.Actions> <ReactionButton picker={true} customEmojis={customEmojiPack} /> <Reaction.Avatars event={event} /> </Event.Actions> </EventCard> // Custom emojis render correctly in: // - Event content // - Reaction picker // - Reaction display // - Avatar tooltips // All connected. All consistent.

Event Cards: Same Pattern, Infinite Flexibility

EventCard.Classic gives you everything. EventCard.Root gives you building blocks. Choose based on how much control you need.

Full Control Composition

<EventCard.Root {event}> <!-- Custom header --> <div class="flex items-center gap-4"> <User.Avatar user={event.author} /> <User.Name user={event.author} /> <CustomTimestamp time={event.created_at} /> </div> <!-- Use default content rendering --> <Event.Content {event} /> <!-- Custom actions --> <Event.Actions> <CustomShareButton /> <ReactionButton {event} /> <CustomBookmark /> </Event.Actions> </EventCard.Root>

Override Just One Piece

<EventCard.Classic {event}> <!-- Keep default header and actions --> <!-- Only customize content rendering --> <Event.Content {event} renderMention={CustomMention} renderHashtag={CustomHashtag} renderLink={CustomLink} /> </EventCard.Classic> // Everything else: default and working // This one thing: exactly how you want it

Primitives Compose Up, Not Just Down

Start with User.Avatar. It works in User.Root. Which works in UserCard.Classic. Which works in FollowPack.Hero. Every layer adds value without breaking the layer below.

Layer 1: Atomic Primitive
<User.Avatar user={user} size="md" />
↓
Layer 2: Composed Primitive
<User.Root user={user}>
  <User.Avatar />
  <User.Name />
  <User.Nip05 />
</User.Root>
↓
Layer 3: Pre-composed Component
<UserCard.Classic user={user} followAction={followAction} />
↓
Layer 4: Complex Block
<FollowPack.Hero users={users} title="Suggested Follows" />

The Magic: You Can Enter At Any Layer

// Need just an avatar? Use the primitive <User.Avatar {user} /> // Need a card? Use the pre-built <UserCard.Classic {user} /> // Need a custom layout? Compose primitives <User.Root {user}> <User.Avatar size="xl" /> <User.Name class="text-3xl" /> </User.Root> // Need to customize the pre-built? Override pieces <UserCard.Classic {user}> <User.Avatar shape="square" /> </UserCard.Classic> // Every approach works. Mix and match.

This Is What Composability Means

Not "here are 100 components, good luck combining them." But "here are primitives that compose naturally, pre-built components that accept custom children, and the freedom to mix both approaches."

Real Example: Custom Event Card With Standard Actions

<EventCard.Root {event}> <!-- Totally custom header --> <YourCustomHeader {event} /> <!-- Use the standard content renderer --> <!-- But swap mention rendering --> <Event.Content {event} mentionComponent={YourMentionStyle} /> <!-- Use standard actions --> <Event.Actions> <ReplyButton {event} /> <ReactionButton {event} picker={true} /> <ZapButton {event} /> </Event.Actions> </EventCard.Root> // Custom where you want it. Standard where you don't. // Everything plays nice together.
